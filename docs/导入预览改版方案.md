# 导入预览改版方案

## 1. 现状分析与问题诊断

当前系统的资产导入流程采用“弹窗预览 + 严格校验”模式，存在以下核心痛点：

### 1.1 校验逻辑过于严苛

- **成本价（Cost Basis）限制**：当前代码（`excelImport.ts`）强制要求成本价必须大于等于 0 且不能为空。然而，在手动添加资产的 UI 中，成本价是可选的。用户在 Excel 中留空或填写 `-` 会被解析为 `null` 并触发报错，导致整行无法导入。这种不一致性让用户感到困惑，因为数据在业务逻辑上是有效的。
- **代码识别限制**：如果 Excel 中的代码不在系统内置库中，预览会直接标记为错误并拒绝导入。虽然这保证了数据的准确性，但对于拥有“非主流”资产的用户来说，这种“一刀切”的拒绝显得不够友好。

### 1.2 交互体验受限

- **只读预览**：目前的预览弹窗仅供查看。如果 100 行数据中有 1 行报错，用户必须关闭弹窗，修改 Excel 文件，重新上传。这种“全有或全无”的逻辑在处理批量数据时效率极低。
- **弹窗空间不足**：在 Modal 中展示复杂的表格和错误列表显得拥挤，难以承载后续规划的“在线修正”功能。

### 1.3 业务逻辑对齐

- **导入策略**：根据 `docs/网页版prd.md` 的既定方向，系统将采用**“覆盖当前持仓，保留历史快照”**的策略。这意味着导入操作将替换现有的所有持仓记录，而非简单的追加。当前的预览逻辑尚未体现这一重大转变。

---

## 2. 改版目标：从“校验器”转向“编辑器”

将导入预览从一个**静态的报错展示框**转变为一个**临时的在线工作区**，为用户提供更灵活的数据处理能力。

### 2.1 核心转变

- **容器升级**：从 Modal 弹窗改为独立的二级页面（如 `/import/preview`）。
- **校验优化**：区分“致命错误”与“非致命警告”，允许在特定条件下继续流程。
- **行级操作**：支持在预览界面直接删除或忽略特定行，减少反复修改 Excel 的次数。

---

## 3. 详细设计方案

### 3.1 校验规则调整建议

| 场景           | 当前行为 | 建议调整 (MVP) | 理由                                               |
| :------------- | :------- | :------------- | :------------------------------------------------- |
| **成本价为空** | 报错拒绝 | **允许为空**   | 与手动添加逻辑保持一致，缺失成本价不应阻断导入。   |
| **成本价为负** | 报错拒绝 | 保持报错       | 负成本价在业务上无意义，属于致命错误。             |
| **未知代码**   | 报错拒绝 | **保持阻断**   | MVP 阶段确保数据 100% 可识别。未来可支持在线关联。 |
| **数量无效**   | 报错拒绝 | 保持报错       | 数量是持仓的核心，必须为正数。                     |

### 3.2 容器升级：二级页面 vs 弹窗

**建议：迁移至独立二级页面。**

**权衡分析：**

- **优势**：提供完整的屏幕空间展示多列数据；支持更复杂的交互（如行删除、过滤、批量操作）；为未来的“在线修正”预留扩展性。
- **代价**：增加了路由跳转的开销；需要处理页面刷新导致的数据丢失问题（可通过状态管理或临时存储解决）。

### 3.3 导入策略适配

- **全量覆盖**：明确告知用户，确认导入后将清空当前所有持仓，并以预览列表中的数据为准。
- **历史保护**：在执行覆盖的同时，系统会自动触发一次净值快照，确保历史曲线的连续性，不会因为持仓清空而出现断裂。

---

## 4. 实施路线图

### 第一阶段：基础框架与逻辑软化 (MVP)

- [ ] **路由迁移**：新增 `/import/preview` 页面，作为导入流程的必经之路。
- [ ] **校验调整**：修改 `excelImport.ts`，允许 `costBasis` 为空（解析为 `null` 或 `undefined`）。
- [ ] **行级管理**：在预览页支持“删除”操作，允许用户剔除无法修正的错误行后继续导入。
- [ ] **覆盖逻辑实现**：按照 PRD 要求，实现“先清空、后插入、保历史”的后端接口。

### 第二阶段：在线修正与增强 (Later Phase)

- [ ] **行级编辑**：支持直接在表格中修改数量、成本价等关键字段。
- [ ] **未知代码关联**：针对系统不识别的代码，提供搜索框让用户手动关联到已有的资产库。
- [ ] **批量忽略**：一键忽略所有包含警告或错误的行。

---

## 5. 决策清单 (User Review)

在开始开发前，请确认以下事项：

1. **是否接受 MVP 阶段仍阻断未知代码？** (建议：是，先保证数据准确性，后期再加关联功能)
2. **是否确认采用“覆盖模式”作为唯一导入方式？** (建议：是，符合 PRD 设定的“事实来源”原则)
3. **是否接受页面跳转带来的交互变化？** (确认用户从点击“上传”到进入预览页的流程跳转)

---

_本方案旨在通过更合理的校验逻辑和更强大的交互容器，提升用户批量管理资产的成功率。_
