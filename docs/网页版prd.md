# 网页版资产追踪器产品需求文档 (PRD)

## 1. 背景与问题陈述

目前资产追踪器（Asset Tracker）的网页版已经具备了基础的资产管理、实时价格获取和投资组合可视化功能。随着用户资产种类的增加，手动录入变得繁琐，用户需要一种更高效的方式（如 Excel/CSV 上传）来批量更新其持仓数据。

然而，引入上传功能面临一个核心决策：**上传的数据应该如何与现有数据合并？**

- 如果是简单的“追加”，会导致重复持仓。
- 如果是“覆盖”，可能会破坏用户辛苦积累的历史净值曲线（Line Chart）。

本 PRD 旨在定义网页版上传功能的行为准则，确保在提供便捷性的同时，保护历史数据的完整性和图表的可读性。

## 2. 目标与非目标

### 目标

- 实现端到端的 Web 端数据上传流程。
- 解决上传数据与现有持仓的冲突问题。
- 确保上传操作不会导致历史净值图表（Portfolio Value History）出现断裂或清空。
- 提供清晰的用户引导和风险提示。

### 非目标

- 本阶段不涉及 OpenClaw 本地 Skill 的实现。
- 不支持复杂的增量合并逻辑（如：仅更新数量变化的资产）。
- 不支持多账户同步。

## 3. 核心决策：覆盖式上传 (Overwrite Strategy)

经过评估，我们选择 **“当前持仓覆盖，历史数据保留”** 的策略。

### 决策理由

1. **数据一致性**：Excel 通常被用户视为“事实来源”（Source of Truth）。覆盖式上传能确保网页端的数据与用户的 Excel 完全同步，避免手动删除旧数据的麻烦。
2. **操作简单**：相比复杂的增量合并（需要处理资产重命名、删除等逻辑），全量覆盖的逻辑更清晰，用户心理预期更明确。
3. **保护历史**：通过将“当前持仓”与“历史快照”在数据库层面分离，我们可以做到在更新持仓的同时，不破坏已有的历史曲线。

## 4. 功能详述

### 4.1 当前持仓 vs 历史快照分离

- **当前持仓 (Current Holdings)**：存储在 `holdings` 表中。这是用户当前拥有的资产列表。
- **历史快照 (Historical Snapshots)**：存储在 `portfolio_value_history` 表中。这是过去每个时间点的资产总值记录。
- **规则**：上传操作仅作用于 `holdings` 表，严禁清空或修改 `portfolio_value_history` 表中的既有记录。

### 4.2 覆盖上传工作流

1. **入口**：在持仓列表页提供“导入 Excel/CSV”按钮。
2. **上传与解析**：用户选择文件，前端解析数据并进行格式校验。
3. **风险确认**：系统弹出强制确认框，文案建议：
   > “导入操作将**替换**您当前的全部持仓数据。您的历史净值图表将不受影响并继续累积。建议在操作前导出备份。”
4. **执行覆盖**：
   - 开启数据库事务。
   - 删除当前用户在 `holdings` 表中的所有记录。
   - 插入解析后的新持仓数据。
   - 提交事务。
5. **自动快照（可选但推荐）**：覆盖完成后，立即触发一次新的价格获取和净值计算，并在 `portfolio_value_history` 中记录一个新点，确保图表能立即反映上传后的变化。

### 4.3 历史图表保护规则

- **禁止清空**：任何上传操作都不得调用 `DELETE FROM portfolio_value_history`。
- **连续性维护**：即使上传后的总资产与上传前差异巨大，图表也应如实反映这种跳变，而不是重置图表。
- **时间戳对齐**：上传产生的快照应记录当前时间，确保与之前的历史点形成连续的时间序列。

## 5. 用户交互与文案建议

- **按钮文案**：`导入数据 (覆盖当前)`
- **成功提示**：`导入成功！已更新 X 项持仓。您的历史记录已安全保留。`
- **错误提示**：`导入失败：文件格式不正确或缺少必要列（资产名称、数量）。`

## 6. 风险与规避

- **风险**：用户误操作导致当前持仓丢失。
  - **规避**：在确认框中增加“导出当前数据作为备份”的快捷链接。
- **风险**：上传频率过高导致历史快照数据冗余。
  - **规避**：限制每日自动快照的频率，或在上传时判断如果当日已有快照则进行更新而非新增。

## 7. MVP 实施范围

1. **前端**：
   - 增加上传组件。
   - 实现简单的 CSV/Excel 解析逻辑。
   - 增加覆盖确认对话框。
2. **后端**：
   - 增加 `replaceHoldings` tRPC 接口。
   - 确保该接口在事务中运行，先删后增。
   - 确保不触碰历史记录表。
3. **验证**：
   - 上传后，持仓列表更新。
   - 刷新页面后，历史趋势图表依然保留上传前的曲线。

## 8. 未来演进

在 Web 端流程跑通后，该逻辑将直接迁移至 OpenClaw Skill 版本。Skill 版本将进一步强化“本地隐私”属性，但核心的“覆盖持仓、保留历史”逻辑保持一致。
