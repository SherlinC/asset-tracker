# 游客 / 登录身份功能补齐计划

## 目标

把下面两类能力同时补齐到两种身份：

1. 游客身份、登录身份都能看到可用的资产折线图
2. 游客身份、登录身份都能使用“趣味转换 / 坐吃山空”面板
3. 两套体验在刷新、导入、编辑持仓后表现一致
4. 全程不删除你已有的真实账号数据，也不覆盖你历史数据库记录

---

## 数据安全前提

这一项必须先定死，后续所有开发都遵守：

- 不删除现有 MySQL 中任何 `users`、`holdings`、`portfolio_value_history` 相关历史数据
- 不对现有登录用户历史曲线做 destructive migration
- 游客数据继续放本地（`localStorage`），与登录数据隔离
- 如果后续需要游客数据升级或迁移，只做“复制式迁移”，不做“覆盖式迁移”
- 新增本地存储结构时带版本号，避免把旧游客数据直接搞坏
- 所有涉及折线图和趣味转换的改动，先加兼容层，再替换页面接线

---

## 当前现状判断

### 已经有的

- 游客模式已经能进入仪表盘
- 游客持仓已经可以走本地存储
- 仪表盘折线图已经开始接入游客本地历史

### 还没补齐的关键点

- “坐吃山空 / Spendover”页面现在主要还是按登录用户路径取数据
- 游客与登录用户的数据来源还没有统一成一套清晰的数据接口
- 游客本地历史、登录历史、刷新逻辑、图表逻辑还有分叉
- 后续如果继续加时间范围、刷新、导入、趣味转换，很容易再次分叉

---

## 推荐实施顺序

建议按 6 个大步骤推进，不要跳步。

## 第 1 步：做一层统一的数据视图层

### 目标

把“游客本地数据”和“登录数据库数据”统一成一套前端可消费的 view model。

### 主要工作

- 抽一个统一的 portfolio data adapter / facade
- 对外统一暴露：
  - `summary`
  - `holdings`
  - `history`
  - `refresh`
  - `mode`（guest / authenticated）
- Dashboard、Noodle、折线图组件只消费统一数据，不直接分散判断 guest / login

### 产出

- 一个统一的 hooks 或 service 层
- 页面不再自己拼 guest/login 分支

### 风险控制

- 先保留旧接口，逐页切换
- 不改数据库结构

### 预估 token

`12k - 20k`

---

## 第 2 步：把折线图能力正式补齐到两种身份

### 目标

保证游客和登录用户都能稳定看到折线图，并且时间范围、空态、刷新表现一致。

### 主要工作

- 统一折线图输入数据结构
- 明确游客历史来源：本地历史快照
- 明确登录历史来源：数据库 `portfolio_value_history`
- 统一时间范围处理：`24h / 7d / 30d / 1y / all`
- 统一空数据文案和 loading 状态
- 修掉游客刷新后图表不更新 / 更新延迟的问题

### 产出

- 一个真正身份无关的折线图消费方式
- 游客第一次导入、编辑、删除、刷新后都能看到曲线变化

### 风险控制

- 登录用户图表逻辑优先保持兼容
- 游客历史条数做上限控制，防止本地存储无限膨胀

### 预估 token

`10k - 18k`

---

## 第 3 步：把“坐吃山空 / 趣味转换”补齐到两种身份

### 目标

让 Noodle / Spendover 页面不再只偏向登录路径，而是游客也能完整可用。

### 主要工作

- Noodle 页面改为消费统一的数据视图层
- “总资产 -> 面条碗数 / 生存时间 / 压力值”统一从 summary + history 计算
- 游客下使用本地 history 做“较昨日变化”或“最近变化”
- 登录下继续使用数据库历史
- 统一刷新按钮行为
- 统一无历史时的 fallback 展示

### 产出

- 游客和登录身份都能打开并正常使用趣味转换面板
- 两种身份都能看到趋势变化，不再只有静态总额

### 风险控制

- 不删除现有登录用户历史线
- 不把游客数据写回数据库

### 预估 token

`12k - 22k`

---

## 第 4 步：统一“导入 / 新增 / 编辑 / 删除 / 刷新”后的联动行为

### 目标

保证所有会影响资产值的操作，都会同步更新：

- 持仓列表
- 总资产摘要
- 折线图
- 趣味转换面板

### 主要工作

- 定义统一的“资产数据变化事件”或刷新策略
- 游客模式下更新本地 holdings 后，自动重算 summary/history/noodle
- 登录模式下 mutation 成功后，统一 invalidate/query refresh
- 避免重复刷新、并发刷新、重复记历史快照

### 产出

- 用户操作后所有面板一起变，不会出现一个更新一个不更新

### 风险控制

- 重点防止重复写历史记录
- 重点防止游客模式触发服务端持久化写入

### 预估 token

`10k - 18k`

---

## 第 5 步：补测试、迁移保护和数据安全保护

### 目标

把“能跑”变成“可长期维护”。

### 主要工作

- 增加 guest / login 双路径测试
- 增加图表数据构建测试
- 增加 Noodle 转换计算测试
- 增加 localStorage schema version / migration 测试
- 增加“不会误删登录用户历史数据”的保护说明和测试

### 产出

- 关键回归测试集
- 游客本地数据结构的版本化方案

### 风险控制

- 所有 migration 必须是向前兼容
- 禁止测试中出现清空生产数据式逻辑

### 预估 token

`8k - 16k`

---

## 第 6 步：收口文档与体验优化

### 目标

把用户能感知到的差异说清楚，把维护成本降下来。

### 主要工作

- 在 UI 上明确标识“游客数据仅保存在本机”
- 可选增加：
  - 游客数据导出
  - 游客数据清空
  - 登录后导入游客本地数据
- 写开发文档：guest vs login 数据流
- 写回归检查清单

### 产出

- 一套完整的使用说明和维护说明
- 后续继续加页面时，不容易把 guest/login 再做散

### 风险控制

- “清空游客数据”必须只清本地，不碰登录库数据
- “导入游客数据到账号”必须做显式确认

### 预估 token

`6k - 12k`

## 推荐优先级

如果希望尽快把“游客也能完整体验”做出来，建议顺序固定为：

1. 第 1 步：统一数据视图层
2. 第 2 步：折线图双身份补齐
3. 第 3 步：坐吃山空双身份补齐
4. 第 4 步：联动统一
5. 第 5 步：测试补齐
6. 第 6 步：体验收口

---

## 里程碑定义

### 里程碑 A

- 游客 / 登录都能在 Dashboard 看折线图
- 不删旧数据

### 里程碑 B

- 游客 / 登录都能在 Spendover 看趣味转换结果
- 导入/编辑后会联动更新

### 里程碑 C

- 双身份路径测试补齐
- 数据安全保护完成

---

## 开工建议

建议下一轮直接从 **第 1 步 + 第 2 步** 一起做。

原因：

- 这是后面所有页面复用的地基
- 如果先单独补 Noodle，很容易又写出第二套 guest/login 分支
- 先把折线图数据抽象统一，后面趣味转换就会简单很多

---

## 明确承诺

后续如果按这份计划实施，我会遵守下面这条：

**不删除你曾经的登录数据，不主动清空数据库历史，不把游客本地数据覆盖进真实账号历史，除非你明确要求。**
